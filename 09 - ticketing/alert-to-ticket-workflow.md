# Alert-to-Ticket Workflow

The SOC Detection Lab demonstrates how security alerts generated by detection rules can automatically create incident tickets.

This workflow simulates how alerts move through a real Security Operations Center (SOC) pipeline.

---

# SOC Alert Pipeline

The complete monitoring pipeline in the lab environment follows this sequence:

```
Attack Simulation
        │
        ▼
Endpoint Telemetry
        │
        ▼
Elastic Agent Log Collection
        │
        ▼
Elastic Security Detection Rules
        │
        ▼
Alert Generated
        │
        ▼
Webhook Triggered
        │
        ▼
osTicket Incident Created
```

This workflow ensures that suspicious activity is automatically tracked within the incident management system.

---

# Step 1 — Attack Simulation

Security events are generated through simulated attack scenarios.

Examples include:

- SSH brute-force attacks
- RDP brute-force attacks
- Mythic command-and-control activity

These actions generate authentication failures or suspicious process events on monitored endpoints.

---

# Step 2 — Log Collection

Elastic Agents installed on the endpoints collect security telemetry.

Examples of collected data include:

```
authentication logs
process execution events
system security logs
```

The logs are forwarded to the Elastic Stack for analysis.

---

# Step 3 — Detection Rules

Elastic Security detection rules analyze incoming telemetry to identify suspicious patterns.

Example detections include:

- repeated authentication failures
- suspicious executable launches
- potential command-and-control activity

When rule conditions are met, Elastic generates an alert.

---

# Step 4 — Alert Generation

Once a rule triggers, an alert appears in the Elastic Security dashboard.

The alert contains contextual data including:

```
rule name
severity level
source IP
target user
affected host
event count
```

These fields provide the initial information needed for investigation.

---

# Step 5 — Webhook Trigger

After the alert is generated, the webhook action sends a request to the osTicket API.

This request contains alert metadata and incident details.

The webhook automatically creates a ticket within the ticketing system.

---

# Step 6 — Ticket Creation

The osTicket API receives the webhook request and generates a new incident ticket.

The ticket contains information about the alert including:

- detection rule name
- source IP address
- targeted username
- affected host
- link to the Kibana alert

Security analysts can then review the ticket and begin investigating the incident.

---

# SOC Workflow Benefits

Automating the alert-to-ticket workflow provides several advantages:

- faster incident tracking
- reduced manual effort
- improved alert visibility
- structured investigation documentation

This integration demonstrates how detection systems and incident management platforms work together in real SOC environments.
