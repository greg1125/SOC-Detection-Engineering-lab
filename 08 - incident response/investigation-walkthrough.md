# Incident Investigation Walkthrough

This section demonstrates how alerts generated by the SOC Detection Lab are investigated by a security analyst.

The investigation process begins when a detection rule triggers an alert within Elastic Security. Analysts must then determine whether the alert represents legitimate malicious activity or a benign event.

The examples below illustrate how alerts generated during the attack simulations were analyzed using the Elastic Security interface.

---

# Investigation Workflow

When an alert is triggered, the investigation typically follows several steps:

1. Review the alert summary
2. Examine event metadata
3. Identify the affected host
4. Determine the source of the activity
5. Analyze related events
6. Assess the severity of the incident
7. Escalate or close the alert

These steps help analysts determine whether the activity represents an actual security incident.

---

# Example Investigation: RDP Brute Force Attack

During the threat simulation phase, repeated authentication attempts were launched against the Windows endpoint using Ncrack.

The RDP brute-force detection rule triggered an alert when the configured threshold of failed login attempts was exceeded.

---

## Step 1 — Review Alert Details

The first step is to review the alert generated by the detection rule.

Important fields include:

```
kibana.alert.rule.name
kibana.alert.severity
source.ip
user.name
host.hostname
kibana.alert.threshold_result.count
```

Example:

```
Rule: RDP-Brute-Force-Attempt
Severity: Medium
Source IP: <ATTACKER_IP>
Target User: <TARGET_USERNAME>
Target Host: <WINDOWS_ENDPOINT>
Failed Attempts: <FAILED_LOGIN_COUNT>
```

These fields immediately indicate that multiple failed login attempts occurred.

---

## Step 2 — Identify the Source of Activity

Next, analysts determine where the login attempts originated.

The `source.ip` field identifies the attacking host.

Example:

```
source.ip: <ATTACKER_IP>
```

In this lab environment, the source system corresponds to the Kali Linux attacker machine used during the attack simulation.

---

## Step 3 — Review Related Events

Analysts then pivot into the event timeline to review related authentication events.

In Elastic Security, analysts can open the alert in **Discover** or **Timeline** to view the surrounding events.

Relevant Windows event codes include:

```
4625 — Failed login attempt
4624 — Successful login
```

Repeated event code `4625` entries indicate that automated login attempts occurred.

---

## Step 4 — Identify the Targeted Account

The `user.name` field reveals which account was targeted during the attack.

Example:

```
user.name: <TARGET_USERNAME>
```

Attackers frequently target administrative accounts, making this field particularly important for investigation.

---

## Step 5 — Determine Attack Pattern

If many authentication attempts occur within a short time window, the activity likely represents an automated brute-force attack.

Indicators include:

- repeated failed login attempts
- attempts targeting the same account
- login attempts occurring within seconds of each other

These patterns are typical of automated credential attacks.

---

## Step 6 — Assess Severity

The severity of the alert helps determine the priority of the investigation.

Example:

```
Severity: Medium
Risk Score: 47
```

Brute-force attacks may indicate an attacker attempting to gain access to the system, so analysts must verify whether any login attempts were successful.

---

## Step 7 — Escalation

If malicious activity is confirmed, the incident should be escalated through the appropriate response workflow.

Possible response actions include:

- blocking the attacking IP address
- disabling targeted accounts
- increasing monitoring of the affected host
- initiating a deeper forensic investigation

# Mythic C2 Investigation Example

A second scenario involved detecting a suspicious executable associated with a Mythic command-and-control payload.

The detection rule identified a suspicious process creation event using Sysmon telemetry.

---

## Suspicious Process Execution

The alert contained the following fields:

```
event.code: 1
event.action: Process Create
process.name: <SUSPICIOUS_EXECUTABLE>
```

Event code `1` corresponds to **process creation events captured by Sysmon**.

This indicates that a new executable was launched on the monitored system.

---

## Process Analysis

Analysts should review the following fields to determine whether the process is malicious:

```
process.name
process.command_line
winlog.event_data.ParentImage
winlog.event_data.Image
```

Suspicious indicators include:

- executables running from temporary directories
- unexpected command-line arguments
- processes spawning from unusual parent processes

---

# Investigation Outcome

Because the activity was intentionally generated during the lab simulation, the alerts were confirmed as expected detections.

However, the investigation process mirrors how real SOC analysts would triage and analyze suspicious activity in a production environment.

This investigation workflow demonstrates how detection rules, telemetry, and alert context combine to support effective security monitoring.
