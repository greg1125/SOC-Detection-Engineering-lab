

# Alert Fields Explained

Elastic Security alerts contain structured fields that provide context about suspicious activity detected within the environment.

These fields allow analysts to quickly determine what occurred, where the activity originated, which systems were affected, and which detection rule triggered the alert.

The examples in this lab are derived from alerts generated by the following detection rules:

- RDP Brute Force Detection
- SSH Brute Force Detection
- Mythic Command-and-Control Detection


---

# Key Alert Fields

## source.ip

The `source.ip` field identifies the system that originated the activity.

In brute-force attack scenarios, this field represents the attacking host performing repeated authentication attempts against a target system.

Example:

```
source.ip: 192.168.199.1
```

This value allows analysts to identify the origin of suspicious activity and determine whether the traffic is coming from:

- an internal host
- a compromised machine
- an external attacker

In this lab environment, the attacking system is the Kali Linux machine used to simulate brute-force attacks.

---

## user.name

The `user.name` field identifies the account associated with the authentication attempt.

Example:

```
user.name: Administrator
```

This field is important for determining which account was targeted during an attack.

Attackers often attempt to compromise:

- administrative accounts
- service accounts
- commonly used usernames

Repeated login attempts against privileged accounts may indicate a credential brute-force attack.

---

## host.name / host.hostname

The `host.name` or `host.hostname` field identifies the system where the event occurred.

Example:

```
host.hostname: WIN-ENKC282BMR5
```

This field tells analysts which system generated the security event.

For example:

- a Windows server receiving RDP authentication attempts
- a Linux endpoint receiving SSH login attempts
- a workstation executing suspicious processes

---

## agent.name

The `agent.name` field identifies the Elastic Agent installed on the endpoint that generated the log data.

Example:

```
agent.name: linux-endpoint
```

Elastic Agents collect telemetry from endpoints and forward it to the Elastic Stack for analysis.

This field helps analysts determine which monitored system produced the log event.

---

## event.code

The `event.code` field identifies the type of system event that occurred.

Example:

```
event.code: 4625
```

In Windows Security logs:

- `4625` indicates a **failed authentication attempt**

This event code is commonly associated with brute-force login activity.

---

## event.kind

The `event.kind` field describes the category of event.

Example:

```
event.kind: signal
```

In Elastic Security, a `signal` event indicates that a detection rule has triggered and generated a security alert.

---

## event.action

The `event.action` field describes the specific action that occurred on the system.

Example:

```
event.action: Process Create
```

In the Mythic C2 simulation, this field identifies the creation of a suspicious process on the endpoint.

This allows analysts to investigate potentially malicious program execution.

---

## kibana.alert.rule.name

The `kibana.alert.rule.name` field identifies the detection rule responsible for generating the alert.

Examples:

```
RDP-Brute-Force-Attempt
SSH-Brute-Force-Attempt
MyDFIR-Mythic-C2-Apollo-Agent-Detected
```

This field tells analysts exactly which detection rule identified the suspicious activity.

---

## kibana.alert.severity

The `kibana.alert.severity` field represents the severity level assigned to the alert.

Examples:

```
medium
critical
```

Severity levels help analysts prioritize investigations.

Typical severity classifications include:

| Severity | Meaning |
|--------|--------|
| Low | Suspicious activity requiring monitoring |
| Medium | Potential malicious activity |
| High | Likely malicious behavior |
| Critical | Confirmed or highly suspicious compromise |

In this lab:

- Brute-force attacks generate **medium severity alerts**
- Mythic C2 activity generates **critical severity alerts**

---

## kibana.alert.threshold_result.count

For threshold-based detection rules, the `kibana.alert.threshold_result.count` field represents the number of events that triggered the alert.

Example:

```
count: 30
```

This value indicates the number of failed authentication attempts detected within the rule's time window.

For example:

```
30 failed login attempts within 6 minutes
```

This threshold helps identify automated brute-force attacks.

---

# Example Alert Structure

Below is a simplified example of a detection alert generated during the lab.

```json
{
  "host.hostname": "windows-endpoint",
  "source.ip": "attacker-ip",
  "user.name": "Administrator",
  "event.code": "4625",
  "kibana.alert.rule.name": "RDP-Brute-Force-Attempt",
  "kibana.alert.severity": "medium",
  "kibana.alert.threshold_result.count": 30
}
```

This alert indicates that multiple failed authentication attempts occurred against the Windows system, triggering the brute-force detection rule.

---

# Why These Fields Matter

These alert fields provide the essential information analysts need to investigate potential security incidents.

By analyzing these fields, SOC analysts can quickly answer key investigation questions:

- Which system was targeted?
- Which account was involved?
- Where did the attack originate?
- What type of activity occurred?
- Which rule detected the event?

Understanding these fields allows analysts to rapidly triage alerts and determine whether further investigation or incident response is required.
